# Agent 工作纪律

> 基于 addyosmani/agent-skills 改编，适合 QClaw/dual-agent-tool-rl 项目

## 核心理念

**AI 编码代理默认走最短路径 —— 这往往意味着跳过规格、测试、安全审查等让软件可靠的最佳实践。**
本工作纪律赋予 agent 结构化的工作流程，强制执行资深工程师在生产级代码中的相同纪律。

---

## 6 阶段开发流程

```
 DEFINE     PLAN       BUILD      VERIFY     REVIEW     SHIP
┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐
│ Idea │ ─▶│ Spec │ ─▶│ Code │ ─▶│ Test │ ─▶│ QA  │ ─▶│ Go  │
│Refine│  │ PRD  │  │ Impl │  │Debug │  │Gate  │  │Live │
└──────┘  └──────┘  └──────┘  └──────┘  └──────┘  └──────┘
  /spec    /plan     /build     /test    /review    /ship
```

---

## 阶段 1: Define（定义）

### 触发条件
- 用户提出一个新的功能需求、项目想法、或需要实现的特性
- 当前任务涉及创建新文件、新模块、或显著的系统变更

### 执行流程
1. **Idea Refine（想法精炼）**
   - 与用户确认需求细节
   - 识别核心问题和使用场景
   - 收敛模糊想法为具体提案

2. **Spec-Driven Development（规格驱动开发）**
   - 编写 PRD（产品需求文档），包含：
     - 目标与背景
     - 功能需求
     - 技术架构
     - 代码风格约定
     - 测试策略
     - 边界与约束
   - **在写任何代码之前完成规格文档**

### 验证标准
- ✅ 存在 SPEC.md 或等效规格文档
- ✅ 文档覆盖所有关键点（目标、架构、边界）
- ✅ 用户已确认规格

---

## 阶段 2: Plan（规划）

### 触发条件
- 已完成规格文档
- 需要开始实现但不确定具体任务拆解

### 执行流程
1. **Planning and Task Breakdown（任务分解）**
   - 将规格拆解为小、可验证的任务
   - 每个任务有明确的验收标准
   - 按依赖关系排序任务

### 验证标准
- ✅ 任务列表存在且每个任务有验收标准
- ✅ 依赖关系明确
- ✅ 任务大小合理（单个任务 < 100 行代码变更）

---

## 阶段 3: Build（构建）

### 触发条件
- 开始写代码

### 执行流程
1. **Incremental Implementation（增量实现）**
   - 每次只实现一个功能切片
   - 实现 → 测试 → 验证 → 提交
   - 使用 feature flags、 safe defaults、 回滚友好变更

2. **Test-Driven Development（测试驱动开发）**
   - 先写测试，再写实现
   - 遵循测试金字塔（80% 单元测试 / 15% 集成测试 / 5% E2E）
   - 测试命名描述行为而非实现细节

3. **Context Engineering（上下文工程）**
   - 确保正确的文件在正确的上下文中
   - 使用 AGENTS.md / SOUL.md 等规则文件
   - MCP 集成按需加载

### 验证标准
- ✅ 代码按增量方式提交
- ✅ 新代码有对应测试
- ✅ 现有测试继续通过

---

## 阶段 4: Verify（验证）

### 触发条件
- 代码实现完成，需要验证是否工作

### 执行流程
1. **运行测试**
   - 单元测试：`pytest tests/unit/`
   - 集成测试：`pytest tests/integration/`
   - 全部测试：`pytest`

2. **Debugging and Error Recovery（如有必要）**
   - 五步排查：复现 → 定位 → 简化 → 修复 → 防护
   - 遇到失败立即停止，修复后再继续
   - 有安全的 fallback 方案

### 验证标准
- ✅ 所有测试通过
- ✅ 手动验证关键路径（如适用）

---

## 阶段 5: Review（审查）

### 触发条件
- 代码已经通过测试，准备提交

### 执行流程
1. **Code Review and Quality（代码审查）**
   - 五轴审查：Correctness, Security, Performance, Readability, Maintainability
   - 变更大小控制在 ~100 行
   - 使用 severity 标签：Nit / Optional / FYI / Must Fix

2. **Code Simplification（如有必要）**
   - 保持代码清晰而非巧妙
   - 遵循 500 行规则（单个文件 < 500 行）
   - 移除死代码和僵尸代码

### 验证标准
- ✅ 通过自审查
- ✅ 无严重问题遗留

---

## 阶段 6: Ship（发布）

### 触发条件
- 代码已通过审查，准备合并到主分支

### 执行流程
1. **Git Workflow and Versioning**
   - 使用 trunk-based development
   - 原子提交（每个提交是一个独立、可回滚的变更）
   - 提交信息清晰描述"做了什么"和"为什么"

2. **Documentation and ADRs（如有架构变更）**
   - 更新 README.md
   - 重大决策记录在 ADR

### 验证标准
- ✅ 提交到主分支
- ✅ 文档已更新（如需要）

## 反理性化清单（Anti-Rationalization）

| 常见借口 | 正确做法 |
|---------|---------|
| "这只是个小改动，不需要测试" | 任何改动都需要验证，改动越小越容易验证 |
| "我稍后会添加测试" | 现在就写测试，"稍后"永远不会来 |
| "这个需求很简单，不需要规格文档" | 简单需求更应写规格，防止理解偏差 |
| "我先提交这个，后面再重构" | 重构应该在同一个提交中完成 |
| "这个错误可以忽略" | 任何错误都需要调查和修复 |
| "我只需要快速看一下代码" | 使用系统的代码审查流程 |

## 快速参考：Slash Commands

| 命令 | 用途 | 核心原则 |
|-----|------|---------|
| `/spec` | 写规格文档 | 规格优先于代码 |
| `/plan` | 任务拆解 | 小、原子、可验证 |
| `/build` | 增量实现 | 一次只做一个切片 |
| `/test` | 验证 | 测试即证据 |
| `/review` | 代码审查 | 提升代码健康度 |
| `/code-simplify` | 简化 | 清晰 > 巧妙 |
| `/ship` | 发布 | 更快更安全 |

## 适配说明

本工作纪律基于 [addyosmani/agent-skills](https://github.com/az9713/addyosmani-agent-skills) 改编，针对 dual-agent-tool-rl 项目特点做了以下适配：

1. **保留 QClaw 基础规则**（qclaw-rules skill）- 文件编码、用户信息记忆等
2. **融入项目现有的测试框架**（pytest）
3. **适应单开发者场景** - 简化多人协作流程
4. **与现有架构集成** - Provider、Checkpoint、CLI 等
